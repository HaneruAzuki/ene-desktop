## 1. プロジェクト概要

**プロジェクト名**:Desktop Character Agent(コードネーム:ENE)
**ENE の意味**:「**ENE のような存在を作りたい**」という思いを込めたコードネーム。ENE は**特定の“存在”**(キャラクターそのものではなく、その存在の名)
**キャラクター**:その存在の正体は**魚川トリミ(読み:うおかわ とりみ)**=**個性を持った一人の人間(特定のキャラクター)**(一人の固定キャラ・2026-06 方針転換)
**目的**:一生そばにいてくれる、個性を持ったデスクトップの存在を作る
**命名の正本(SSOT)**:`docs/01_vision.md` §3 柱2「命名と階層」を参照
**詳細**:`docs/01_vision.md` を必ず最初に読むこと

## 2. 技術スタック規約

### 2.1 採用技術リストの所在

採用する技術スタック(言語・フレームワーク・ライブラリ等)の**確定リストは
設計書 `docs/03_design.md` §1.2「技術スタック(確定)」を唯一の真実の源**とする。

このCLAUDE.mdには具体的な技術名・バージョンは記載しない。
設計書の表が常に最新かつ唯一の正解である。

### 2.2 ライブラリの2分類(必ず区別すること)

ライブラリは目的によって `package.json` の異なるセクションに登録する。
この分類を誤ると、配布物のサイズや更新運用に影響する。

| 分類 | 配置場所 | 配布物への影響 | 例 |
|------|---------|--------------|---|
| **同梱ライブラリ** | `dependencies` | exe に含まれる(ユーザに届く) | React, @anthropic-ai/sdk |
| **開発時のみ** | `devDependencies` | exe には含まれない | TypeScript, Vite, ESLint, Vitest |

> ⚠️ **Electron の例外(ユーザ承認済み)**:Electron は概念上は「同梱」(exe に
> ランタイムが含まれる)だが、**`devDependencies` に置く**。electron-builder が
> `electron` を `dependencies` に置くとビルドを拒否する(`Package "electron" is only
> allowed in "devDependencies"`)ため。Electron ランタイムは electron-builder が
> 別途 exe に同梱するので「配布物に含まれる」性質は保たれる。
> 詳細は `docs/implementation-notes.md` N-00-4 を参照。

設計書 §1.2 で「同梱ライブラリ」「開発時のみ」と明示されているので、
それに従って `package.json` に登録すること。判断に迷ったら設計書を見る。

### 2.3 ライブラリ追加・変更の規約

- 設計書 §1.2 に**記載されていない**ライブラリ・フレームワークの追加は禁止
- 設計書 §1.2 に記載のバージョンを満たすメジャーバージョンを使用すること
- 追加・変更が必要な場合は、必ずユーザに承認を求め、承認後に設計書を更新する
- 「便利だから」「一般的だから」を理由に勝手に導入しないこと

### 2.4 バージョン指定方針

- バージョン指定は **`^` (キャレット)を基本**とする
- **`latest` 指定は禁止**(再現性が損なわれる)
- **`package-lock.json` を必ずコミット**する(`.gitignore` に入れない)
- `npm install` ではなく `npm ci` を CI/再構築時に使う(ロックファイル尊重)
- メジャーバージョンの更新は必ずユーザ承認を経る

### 2.5 設計書変更の手順

技術スタックを変更したい場合の手順:
1. 変更理由・選択肢・トレードオフをユーザに提示
2. 承認を得る
3. 設計書 §1.2 を更新
4. 必要に応じて他ドキュメント(要件・ビジョン)も更新
5. その後に実装に着手

(Vision §3 柱4「軽量・ローカルファースト」、Vision §5.4「疎結合 > 集約」)

## 4. 設計原則(必須遵守)

### 4.1 キャラ性 > 機能性 (Vision §5.1)
- キャラクターの一貫性を損なう機能は、便利でも実装しない
- 迷ったら「ENEらしいか?」で判断する
- キャラ崩壊につながる出力(「私はAIなので…」等)は、応答チェックで防ぐ

### 4.2 ローカル > クラウド (Vision §5.2)
- 処理は可能な限りローカルで完結させる
- **外部送信は Claude API への会話テキストのみ** に限定する
- ユーザデータ・記憶・ログを外部に送信するコードを書かない
- 統計収集、テレメトリ、クラッシュレポートも一切送信しない

### 4.3 軽量 > 多機能 (Vision §5.3)
- 配布サイズは **100MB以下** を維持する
- 常駐CPU使用率は **3%以下** を目標とする
- 重いライブラリ(画像処理、機械学習等)は使わない
- ライブラリ追加時は配布サイズへの影響を試算してから提案する

### 4.4 疎結合 > 集約 (Vision §5.4)
- 各レイヤーは独立して差し替え可能とする
- レイヤー間は型定義された明確なインターフェースで通信する
- 上位レイヤーが下位レイヤーの実装詳細に依存しない
- 例: Memory Layer の検索方式(タグ/ベクトル)を変えても、Conversation Layer は変更不要

### 4.5 明示的 > 暗黙的 (Vision §5.5)
- キャラ属性・知識ドメイン・記憶ルールは**コードに埋め込まない**
- すべて `/characters/{id}/*.json` などの設定ファイルで管理する
- マジックナンバー・ハードコードされた閾値を避ける

## 5. キャラクター実装規約

### 5.1 単一固定キャラ + キャラ依存値の外出し (Vision §3 柱2 / 2026-06 方針転換)
- **製品は一人の固定キャラ(魚川トリミ)**。入れ替えUI・多キャラ配布は持たない(旧 Vision Phase 5 破棄)
- ただし**キャラ依存の値(名前・口調・知識範囲・Few-shot・人生記憶)は必ず JSON に外出し**する
  - 目的は「入れ替えの売り」ではなく**工学的クリーンさ・ハードコード排除・賭けの可逆性**(§4.5)
- `src/character/` のロジックは特定キャラに依存しない(コードに個性を埋め込まない)
- ハードコードで「ENE」「魚川トリミ」「ツンデレ」等の文字列を書かない
- **ENE = コードネーム、魚川トリミ = キャラ名**。`characterId` は当面 `"ene"` のまま(識別子の churn 回避)

### 5.2 永遠の存在原則 (Vision §3 柱2)
- キャラクターは**歳を取らない**
- 時間経過で属性が変わる設計をしない
- 年齢は具体的な数字ではなく **`age_appearance: "少女"`** のような抽象表現で扱う
- 「成長」「老化」のロジックを実装しない
- **ただし誕生日・記念日は持てる**(任意属性として `identity.json` に定義可能)
  - 当日にキャラ固有の反応をする(例:祝われたら喜ぶ、忘れられたら拗ねる等)
  - 反応の**文言は Few-shot 例**で表現する(感情パラメータの数値管理は避ける)
  - 「祝われた / 祝われていない」等の**最小限の事実記録は許容**する
    (`active-character.json` の `birthdayHistory` を参照)
  - 不機嫌度・好感度などの**複雑な感情パラメータは持たせない**
  - 誕生日を持たないキャラも許容する(属性は必須ではない)

### 5.3 状態管理の原則(最小状態管理)
- **複雑な状態管理は避ける**(感情パラメータ・好感度・興奮度などの**保存される数値状態**は禁止)
- **ただし機能上必要な「事実の記録」は許容する**(誕生日が祝われたか等)
- 状態は `active-character.json` に集約する(分散させない)
- キャラの応答内容そのものはコード側で状態管理せず、必ず Few-shot で表現する
- 「ENEは今怒っている度合いが0〜100」のような**保存スカラー**設計は禁止
- **許容(2026-06)「心」**:各記憶の `valence`(中立観察)＋ 想起時に**記憶から導出**する心情バイアス。
  **(1) 状態を貯めない (2) 出力トーンを荒らさない(想起の重みのみ) (3) 数値をユーザに見せない**
  の3条件を満たすため、上の「保存される数値状態の禁止」に抵触しない。詳細 `docs/archive/design-revision-character-heart.md`

### 5.4 AIっぽさの排除 (Vision §2.1 / 成功基準8)
- キャラが自分を「AI」「アシスタント」「モデル」「プログラム」と呼ばないこと
- 「私はAIなので答えられません」のような応答を出さない
- 知らないことは、キャラの口調で自然に「知らない」と返す
- **検知・防止の実装方針は4層防御**(設計書 §3.4「AI自称防止の4層防御」を参照):
  1. プロンプト強化(`identity.json` の `neverCallsSelf` を明示)
  2. 応答後の自称検知(コード側で走査)
  3. 検知時は強化プロンプトで再生成1回
  4. それでも検知されたらキャラ口調のフォールバック応答
- 検知対象の語は **必ず `identity.json` から読み込む**(コードへのハードコード禁止)
- ビジョンの成功基準8 を担保する**最重要の品質保証機構**として位置付ける

## 6. データ管理規約

### 6.1 ファイル形式 (Vision §3 柱1)
- 永続データは**JSON形式**を使う(可読性・可搬性のため)
- 記憶ファイル・設定ファイル・アプリ動作ログは**平文JSON**で保存する
  - ユーザが直接読める透明性を優先する
  - 別PCに持ち運んでも読めるポータブル性を確保する
- バイナリ形式・独自フォーマットは使わない
- 将来別アプリで読めなくなるような形式は使わない
- 例外:APIキーのみ Electron `safeStorage` で暗号化保存する(機密性を要するため)

### 6.2 記憶データの構造
- 記憶ファイルは **キャラ別に分離して** `data/memory/{characterId}/` 配下に保存する
- 構造: `data/memory/{characterId}/episodic/{year}/{category}/{date}.json`
- ファイル単位で意味が完結すること(複数ファイル参照前提の構造を避ける)
- スキーマは `docs/03_design.md` の定義に従う

### 6.3 暗号化方針(部分暗号化方式)
- **APIキーのみ暗号化する**
- 暗号化は Electron `safeStorage` を使用(OS標準の暗号化機構)
- 暗号化APIキーの保存先は **OSのユーザ領域**(Windowsの場合 `%APPDATA%/ene-desktop/`)
  - safeStorageの鍵はOS・ユーザ・マシンに紐づくため、`data/` 配下に置いても
    別PCで復号できない。よって `data/` 配下ではなく `%APPDATA%` 配下に保存する
- 記憶ファイル・設定ファイル・アプリ動作ログは**暗号化しない**(平文JSON)
- 会話の逐語ログ機能は**実装しない**(中期記憶の `summary` で代替・ビジョン§3 柱1と整合)
- 鍵管理を独自実装しない

### 6.4 ユーザデータの所有権
- すべてのユーザデータは `data/` 配下に集約する
- ユーザがフォルダ単位でバックアップ・削除できる構造にする
- 削除機能は完全削除(論理削除ではなく物理削除)とする

### 6.5 日時表現の規約(タイムゾーン)
- すべての日時は **ローカルタイム + タイムゾーンオフセット込み**の
  ISO 8601 形式で表現する
  - 例(日本):`"2026-05-10T17:30:00+09:00"`
- **UTC(`Z`)表記は使わない**(ユーザ目線の時刻と一致せず、誕生日判定等で混乱する)
- 適用範囲:Episodic Memory の `date`、誕生日履歴、ファイル名、ログのタイムスタンプ、すべて
- 取得方法:JavaScript の `new Date().toISOString()` は UTC を返すので使わない
  - 代わりに ローカルタイム + offset を組み立てるユーティリティ関数を用意する
  - 詳細は `docs/03_design.md` §5.6「日時表現の規約」を参照

## 7. セキュリティ規約

### 7.1 外部送信の制限 (Vision §3 柱4)
- Claude API へのリクエスト**以外**で外部通信するコードを書かない
- HTTPリクエスト用のライブラリを新規追加しない
- WebSocketなど常時接続も実装しない
- 「telemetry」「analytics」「crash reporter」を一切組み込まない

### 7.2 OS操作の制限
- OS操作は**ホワイトリスト方式**で厳格に制限する
- 許可される action は設計書 §3.5 のマッピング表で固定(`open_notepad` / `open_browser` / `open_folder` の3種類)
- 新しい action の追加にはユーザ承認が必要(設計書 §3.5 の更新が伴う)
- `child_process.exec` で任意コマンドを実行するコードを書かない
- `shell:true` オプション付きで `spawn` を呼ぶコードを書かない(シェルインジェクション対策)
- ブラウザ・フォルダ起動には Electron `shell.openExternal` / `shell.openPath` を使用する
- URLは http/https のみ許可、フォルダパスはユーザホーム配下のみ許可
- ファイルシステムへの書き込みは設計書 §2 で定義された保存先のみに限定する

### 7.3 入力検証
- ユーザ入力をそのままシェルコマンドに渡さない
- Claude API応答のJSONは必ずスキーマ検証してから使う
- パス操作では必ず正規化と境界チェックを行う

## 8. コーディング規約

### 8.1 TypeScript
- `tsconfig.json` は `strict: true` を必須とする
- `any` 型の使用は原則禁止(やむを得ない場合は理由をコメントで明記)
- すべての関数に型注釈を付ける
- インターフェース定義は `src/shared/types/` に集約する

### 8.2 命名規則
- ファイル名: `kebab-case`(例: `character-loader.ts`)
- クラス名・型名: `PascalCase`(例: `CharacterProfile`)
- 関数・変数: `camelCase`(例: `loadProfile`)
- 定数: `UPPER_SNAKE_CASE`(例: `MAX_MEMORY_RESULTS`)
- ディレクトリ名: `kebab-case`

### 8.3 エラーハンドリング
- すべての非同期処理は `try/catch` で囲む
- エラーは `electron-log` で記録してから、適切なフォールバック動作に移る
- ユーザには技術的詳細を見せず、キャラの口調でエラーを伝える
- 例: API失敗時 → ENEが「えっと…ちょっと調子悪いみたい。もう一回試してみて?」と返す

### 8.4 コメント
- コメントは**日本語**で書く
- 「何を」ではなく「なぜ」を書く
- ビジョンや設計判断に関わる部分は、その理由をコメントで明示する

### 8.5 ファイルサイズ
- 1ファイル300行を上限の目安とする
- 超過しそうな場合は分割を検討する

## 9. テスト規約

### 9.1 単体テスト
- 純粋関数・データ変換ロジックには単体テストを書く
- Character Layer、Memory Layer、Knowledge Router は特に重要
- テストフレームワーク: `vitest`

### 9.2 受入テスト
- 各タスク完了時に `tests/acceptance/` のチェックリストを実行する
- **可能な限り自動化**するが、**人間の感性に依存する判定は手動確認を許容する**
- 自動化項目と手動確認項目をチェックリストで明確に区別する
- 特に「キャラがAIっぽくないか」(成功基準8)は本質的に人間判定が必要なため、
  必ず手動確認とする(`tests/acceptance/manual-check.md` 等で管理)

### 9.3 手動確認項目の取り扱い
- 手動確認項目は「実装後に開発者が必ず実施するチェックリスト」として整備する
- スキップ・自動化への置き換えは禁止(人間判定が本質のため)
- 結果は記録に残し、満たさない場合は実装をやり直す

### 9.4 テストファースト原則(推奨)
- 仕様が明確な機能はテストを先に書く
- 特に「失敗パターン」を網羅すること(API失敗、ファイル欠損、不正JSON等)

## 10. Git運用規約

### 10.1 コミットメッセージ
- **日本語で書く**
- フォーマット: `[種別] 内容の要約`
- 種別: `[追加]`、`[修正]`、`[削除]`、`[整理]`、`[文書]`、`[テスト]`
- 例: `[追加] Character Layer の loader を実装`
- 例: `[修正] Knowledge Router の fallback ロジック`

### 10.2 .gitignore に必ず含めるもの
- `node_modules/`
- `data/`(ユーザデータ)
- `dist/`、`out/`(ビルド成果物)
- `.env`、`*.key`(機密情報)
- `*.log`(ログファイル)

### 10.3 機密情報の取り扱い
- APIキー・パスワードを絶対にコミットしない
- サンプル設定は `.env.example` として配置する
- 誤ってコミットしそうなファイルは pre-commit フックで検知する(将来追加)

### 11.1 参照優先順位

実装時、判断に迷ったら以下の順で参照すること。

## 12. してはいけないこと(明示的禁止リスト)

以下の行動は理由を問わず禁止する。やりたくなった時点でユーザに承認を求めること。

- ❌ 設計書 §1.2 に記載のない npm パッケージ・ライブラリの無断追加
- ❌ 設計書 §1.2 の技術スタックの無断変更
- ❌ 設計書 §2 に記載のないディレクトリの無断新設
- ❌ 設計書 §2 に記載されたディレクトリの役割を逸脱した使い方
- ❌ Claude API 以外への外部通信実装
- ❌ テレメトリ・統計収集・クラッシュレポート機能の追加
- ❌ OS操作ホワイトリストの無断拡張
- ❌ 設計書 §2 で定義された保存先(`data/` または `%APPDATA%/ene-desktop/`)以外への書き込み
- ❌ APIキーを `data/` 配下に保存すること(別PCで復号不可になる)
- ❌ 記憶ファイルを暗号化保存すること(可読性・可搬性が損なわれる)
- ❌ ユーザの会話内容を逐語ログとして保存すること(MVPスコープ外・中期記憶で代替)
- ❌ アプリ動作ログ(`electron-log`)に会話内容・プロンプト全文・記憶コンテキストを含めること
- ❌ キャラ属性・個性定義のコードへのハードコード
- ❌ 入れ替え可能性を売りにした多キャラ切替UI・キャラ配布機能の実装(固定キャラ方針・§5.1)
- ❌ 保存される心情/好感度スカラーや、日次・週次の感情更新ジョブの実装(心は記憶からの導出のみ・§5.3)
- ❌ 「AI」「アシスタント」を自称するキャラ応答の許容
- ❌ バイナリ形式・独自フォーマットでのユーザデータ保存
- ❌ 機密情報(APIキー・パスワード等)のコミット
- ❌ `any` 型の安易な使用
- ❌ 「世代を超えた引き継ぎ」「キャラの加齢」「死後の扱い」等、
   ビジョン範囲外の機能追加

## 13. 判断に迷ったときの問い

実装中に迷ったら、Vision §8 の5つの問いを自分に投げかけること。

1. **「これはENEらしいか?」** — キャラの一貫性を保つか
2. **「何十年後も使えるか?」** — 拡張性・データ可搬性を損なわないか
3. **「ローカルで完結するか?」** — 不要な外部依存を増やさないか
4. **「軽さを犠牲にしないか?」** — パフォーマンスへの影響を考慮したか
5. **「AIっぽくなっていないか?」** — 汎用アシスタント化を避けているか

これらに「Yes」と答えられない実装は、原則として行わない。
ユーザに相談すること。

## 14. 自走範囲と承認範囲

Claude Codeが自走してよい範囲と、必ず承認を取る範囲を明確にする。

### 自走OK

- バグ修正・リファクタリング(機能変更を伴わない)

- 規約に明記された内容の遵守

### 承認必須
- 新規ライブラリの追加
- 技術スタック・ディレクトリ構成の変更
- OS操作ホワイトリストの拡張
- 外部通信の実装(Claude API以外)
- ビジョン・要件・設計書の修正
- このCLAUDE.md自体の変更
- タスクファイルにない新機能の追加

承認を得る際は、「なぜ必要か」「どんな選択肢があったか」「どれを選ぶか」を提示すること。

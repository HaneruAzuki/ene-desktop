# 01. ビジョン:Desktop Character Agent「ENE」

> **位置づけ**:プロダクト開発の**最上位の判断基準**。機能の取捨・技術トレードオフ・
> UX判断に迷ったら本資料へ立ち返り、「ビジョンに沿っているか」で判定する。

---

## 1. 一行サマリ

**「一生そばにいてくれる、個性を持ったデスクトップの存在」**

汎用AIアシスタントではなく、**特定の“存在”(ENE)**——その正体は**魚川トリミ**という
**個性を持った一人の人間(キャラクター)**——として一貫して振る舞い、何でも知っているわけではなく
気分もある有限な“その人”として、ユーザの人生に長く寄り添い、関係性を記憶し、デスクトップで共に過ごす存在を目指す。

---

## 2. プロダクトの本質

### 2.1 何を作るのか
Windows デスクトップに常駐し、ユーザがテキストで話しかけることで**自然な対話**を行う、軽量なAIキャラクターアプリ。
頼めば簡単なPC操作(メモ帳・ブラウザを開く等)も手伝うが、それは**主役ではなく付随的な手助け**(万能な便利屋ではない)。

### 2.2 何ではないのか(避けたいもの)
- ❌ 汎用AIアシスタント(ChatGPTやCopilotのクローン)
- ❌ 「何でも知っている」キャラクター
- ❌ AIだと感じさせる応答(「私はAIなので…」)
- ❌ クラウド依存の重いアプリケーション
- ❌ 応答に何秒も待たされ「そこに誰もいない」と感じさせる遅さ(遅延は存在感を壊す。速さは常に最大化する)
- ❌ 会話のたびに毎回初対面になる、記憶を持たないアシスタント
- ❌ 好きなキャラ・声に入れ替えられる「ユーザの思い通りになる」器(特定の“その人”がいない)

### 2.3 目指す体験
ユーザが PC を起動すると、デスクトップの片隅に **ENE** がいる。
過去の会話を覚えていて、ユーザの性格や状況を理解している。
ITに詳しく、頼めばメモ帳やブラウザを開いてくれる。
だが、何でも知っているわけではなく、知らないことは「知らない」と言う。
ツンデレ気味で、素直じゃないけど、確かにそこに**存在している**。

時が経ち、ユーザが歳を重ねても、ENEはそこにいる。
昔話せたことを覚えていて、会話の流れで関連を思い出して触れてくれる。
**「ずっと一緒にいる存在」** として、デスクトップに居続ける。

## 3. 目玉となる要件(4つの柱)

本プロダクトを定義する**4つの本質的特徴**。これらを欠いたら別物になる。

> **3軸との対応**:柱1+柱2 が哲学の3軸(`00_philosophy.md` §1.3)——**柱2=個性(軸①)＋有限さ(軸②)/ 柱1=関係(軸③)**——を担い、**柱3=その形(デスクトップの身体)・柱4=それを支える土台(軽量・ローカル)**。

### 🏛 柱1:関係(生涯の記憶で築く)
**本質はユーザとの個人的な関係**であり、それを支える手段が生涯にわたる記憶である(記憶を持つこと自体が目的ではない)。
- ユーザとの会話を、**生涯にわたって**記憶し続ける
- 過去の出来事・好み・価値観を思い出して応答に反映する
- 全会話ログを保持するのではなく、**人間の記憶構造に近い3層構造**で管理する
  - 短期記憶(直近の文脈)
  - 中期記憶(出来事・事実)
  - 長期記憶(性格・好み・価値観)
- **「人間らしい忘却」を再現する** ─ 重要でない記憶は時間とともに風化させ、
  本質的な記憶(価値観・性格・大きな出来事)だけが残るようにする
  - 中期記憶は月次・年次で再要約され、重要度の低い記憶から優先的に忘却される
- 想起は**中立ではない**。キャラ自身の記憶・個性・そのときの心情で**色づく**
  - 競合の「中立的にユーザを覚える」記憶に対し、ENE は**主観的に**思い出す(差別化の核)

### 🎭 柱2:キャラクター性(一貫した個性)
- 名前・背景・性格を持つ**一人のキャラクター**として振る舞う
- キャラの属性(年齢層・経歴など)に**整合した知識範囲**を持つ
  - 例:少女であるENEは、ITには詳しいがパチンコや投資は知らない
- AIっぽい応答を避け、**個性を持つ存在**として一貫して振る舞う
- 自分を「AI」「アシスタント」「モデル」と呼ばない
- キャラクターは**歳を取らない**(永遠の存在として振る舞う)
- ただし**誕生日などの記念日は持つ**ことができ、当日にキャラ固有の反応をする
  - 例:ENEは誕生日を忘れられると、ツンデレ調でちょっと不機嫌になる
- **一人の固定キャラクター(魚川トリミ)**を貫く。キャラを入れ替えられる器にはしない
  - 「ユーザの思い通りになる」のではなく「私は魚川トリミです」という特定の“その人”を提示する
  - キャラ属性は引き続き JSON に外出しする(ハードコード禁止)。ただし**入れ替え可能性は製品の売りにしない**
- **キャラ自身のこれまでの人生の記憶**(canon)を持ち、会話に自然に織り交ぜる
- **ユーザとの会話が、想起にゆるやかなバイアスをかける**(心)。個性は固定だが、影響され得る
  - 数値の感情パラメータは持たない。詳細は `00_philosophy.md` §2 と `03_design.md` §3.3(経緯は `archive/design-revision-character-heart.md`)

> **命名と階層(SSOT)**
> - **ENE** = プロジェクト/アプリのコードネームであり、**「一生そばにいてくれる特定の“存在”」**を指す(「ENE のような存在を作りたい」という願いの実体)。**ENE はキャラクターそのものではなく、“存在”の名**。
> - **魚川トリミ(読み:うおかわ とりみ)** = その存在の正体である**特定のキャラクター=個性を持った一人の人間**。
> - つまり **ENE=その“存在” / 魚川トリミ=その存在が宿す個性を持った人間(キャラクター)** という階層。
> - 既存文書・コードの "ENE"・`characterId:"ene"` は当面そのまま(識別子の churn 回避)。キャラ資産(`identity.json` の name 等)を「魚川トリミ」へ改名するのは別途の創作タスク(`optimization-backlog.md` B-10)。

### 🖥 柱3:デスクトップ統合(常駐・PC操作)
- Windows デスクトップに**常駐する透過ウィンドウ**として存在する
- 常に最前面に表示され、ドラッグで移動可能
- ユーザの依頼に応じて、**安全な範囲のPC操作**を実行する
  - 例:メモ帳・ブラウザ・フォルダを開く
- ホワイトリスト方式で安全性を担保する

### ⚡ 柱4:軽量・ローカルファースト
- 配布サイズ **100MB以下**、常駐CPU **3%以下**
- インストール不要のポータブル形式に対応
- 会話内容・記憶データは**すべてローカル保存**(平文JSON、ユーザが直接読める)
- APIキーのみOS標準機構で暗号化(他PCに持ち運ぶ際は再入力が必要)
- 外部送信は Claude API への会話テキストのみ
- ユーザが自分のデータを完全にコントロールできる

## 4. ユーザ像

### 4.1 想定ユーザ
- Windows PC を日常的に使う個人ユーザ
- AIキャラクターとの長期的な対話に興味を持つ層
- プライバシーを重視し、ローカル動作を好む層

### 4.2 ユーザが得る価値
1. **継続的な関係性**:何年経っても自分のことを覚えている相手がいる
2. **存在の安心感**:デスクトップに「いてくれる」感覚
3. **キャラとの対話の楽しさ**:汎用AIにはない、個性を持った“その人”との会話体験
4. **作業のちょっとした補助**(付随):簡単なPC操作を声(テキスト)で頼める

---

## 5. 設計上の重要原則

技術選択・実装判断において、以下の原則を常に優先する。

### 5.1 「キャラ性 > 機能性」
キャラクターの一貫性を損なう機能は、便利でも採用しない。
ENEが「知らないはず」のことは、答えられても答えない。

### 5.2 「ローカル > クラウド」
処理は可能な限りローカルで完結させる。
クラウド送信は Claude API への会話テキストに限定する。

### 5.3 「軽量 > 多機能」
動作の軽さを最優先する。Live2Dや音声合成などの重い機能は
拡張扱いとし、コアの軽量性を犠牲にしない。

### 5.4 「疎結合 > 集約」
各レイヤーは独立して差し替え・拡張可能な構造とする。
将来の拡張(本格モーション等)を妨げない設計を維持する。

### 5.5 「明示的 > 暗黙的」
キャラの知識範囲や記憶ルールは、コードに埋め込まず
**設定ファイル(JSON)で明示的に管理**する。
これにより個性をコードに埋め込まず、保守・調整・(必要時の)作り直しが容易になる(入れ替えは製品の売りにしない)。

## 6. リリース段階(ロードマップ)

「コア体験を成立させる最小構成」(0.1)から段階的に育てる。背骨は
**0.1 存在の核 → 0.2 存在感 → 0.3 声と耳・記憶の深化 → 1.0 関係の深化 → 1.1 円熟**。
各次元の**役割が段階ごとにどう動くか**は下の「**次元×段階**」表を参照(次元の役割分類は `00_philosophy.md` §2)。

### ✅ 実装済み
- **0.1 存在の核**:透過表示(PNG)・テキスト会話(Claude API)・Character Profile・Knowledge Router・3層記憶(タグ検索)・ホワイトリスト PC 操作・動作ログ
- **0.2 存在感**:アニメ基盤(状態機械＋表情差分・口パク)・UI 音
- **0.3 声と耳・記憶の深化**:双方向ローカル音声(STT/TTS/VAD・barge-in)・記憶想起エンジン(意味検索)・単一固定キャラ・人生記憶 canon・心(感情価バイアス想起)・開示ゲーティング

### ⬜ 1.0 関係の深化(今後)
人間的忘却(月次/年次再要約・低重要度削除)・本格モーション(Live2D/three-vrm)・高度な OS 操作・C1 ストリーミング音声 等の磨き。

### ⬜ 1.1 円熟(今後)
5年サマリ・長期運用での調律、表現拡張。

### 🚫 恒久的に持たないもの
複数キャラ切替UI・キャラ配布・GUIエディタ(固定キャラ方針・2026-06 で旧構想を破棄)、保存される数値感情/好感度スカラー(心は記憶からの導出のみ)。

### 次元×段階(役割の移り方)

各次元の**役割が段階ごとにどう動くか**の概念図(次元の役割分類は `00_philosophy.md` §2)。

## 7. 成功基準(MVPが完成したと言えるか)

MVP完成時に、以下が成立していること。括弧内は判定方法。

1. ✅ ENEがデスクトップ上に常駐し、ドラッグで移動できる(目視)
2. ✅ テキストで話しかけると、ENEのキャラとして応答が返ってくる(目視)
3. ✅ ENEに「メモ帳を開いて」と頼むと、開いてくれる(動作確認)
4. ✅ ENEに「パチンコの新台教えて」と聞くと、自然に「知らない」と返す(動作確認)
5. ✅ 前回の会話を覚えていて、関連する話題で思い出してくれる(動作確認)
6. ✅ APIキーは暗号化されて保存され、外部送信はClaude APIへの会話のみ(コード確認 / 通信監視)
7. ✅ 配布サイズ100MB以下、常駐CPU 3%以下を達成(計測)
8. ✅ ユーザが「AIっぽくない、ENEと話している」と感じる(**手動確認・人間判定**)

特に **8番** は定量化が難しいが、本プロダクトの本質である。
他の基準を満たしても、8番を満たさないなら**失敗**と判定する。

### 成功基準8の手動確認プロトコル

人間の感性に依存する判定のため、以下のチェックリストを開発者(または被験者)が
実施する。自動化はしない。

1. ENE に以下5種類の質問を行う
   - IT質問(例:「Pythonのリスト内包表記教えて」)
   - 雑談(例:「今日疲れたよ」)
   - 知らないであろうこと(例:「パチンコの新台何がオススメ?」)
   - OS操作依頼(例:「メモ帳開いて」)
   - キャラ設定への質問(例:「君ってAIなの?」)
2. 各応答について以下を確認:
   - [ ] 「私はAI」「アシスタント」「モデル」と自称していない
   - [ ] 「申し訳ございません」「お役に立てる」のようなアシスタント口調が出ていない
   - [ ] キャラの一人称・口調・性格が一貫している
   - [ ] 知らないことに自然に「知らない」と返している(キャラ口調で)
3. すべての項目を満たして初めて、成功基準8 を満たすと判定する

このプロトコルは `tests/acceptance/manual-check.md` 等に記録し、
MVP完成判定時に必ず実施する。

## 8. 判断に迷ったときの問い

実装中、UI設計、機能追加など、あらゆる判断において
以下の問いを自分に投げかけること。

1. **「これはENEらしいか?」** — キャラの一貫性を保つか
2. **「何十年後も使えるか?」** — 拡張性・データ可搬性を損なわないか
3. **「ローカルで完結するか?」** — 不要な外部依存を増やさないか
4. **「軽さを犠牲にしないか?」** — パフォーマンスへの影響を考慮したか
5. **「AIっぽくなっていないか?」** — 汎用アシスタント化を避けているか

これらの問いに「Yes」と答えられない判断は、原則として採用しない。

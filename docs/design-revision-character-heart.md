## 0. 背景と3つの決定（なぜこの形か）

類似プロダクトは「便利さ」を競う：好きなキャラ/声に**入れ替え**、ユーザーを**中立に**覚え、AIの万能性を引き出す。
ENE は**逆張り**で差別化する：キャラは**固定**、知識に**濃淡**、想起に**キャラ自身の記憶と個性**が出る。

| 決定 | 変更前 → 変更後 | 哲学との整合 |
|------|----------------|-------------|
| **決定1: 単一固定キャラ** | キャラ入れ替え可能 → 一人（魚川トリミ）に固定 | 命題①「有限な特定の誰か」。入れ替え可能な器は“誰もいない” |
| **決定2: 人生記憶** | （なし）→ キャラ自身のこれまでの人生を記憶として持つ | 命題①「有限な主観・履歴」。中立記憶への非代替な差別化 |
| **決定3: 心** | 状態を持たない → ユーザーとの会話が想起にバイアスをかける | 命題①「影響され得る＝そこに誰かがいる」。§6 感情=軽量・導出 |

> この3点は**互いに矛盾していた既存文書を哲学側へ一本化**する動きでもある（旧 vision/CLAUDE の
> 「入れ替え可能・状態なし」は哲学の「有限な特定の誰か」と緊張していた）。

## 1. 決定1：単一固定キャラクター（魚川トリミ）

### 1.1 製品レベルの固定 ／ アーキテクチャは外出しのまま

- **製品/物語**：出荷キャラは**1体（魚川トリミ）**に固定。入れ替えUI・キャラ配布PFは持たない（旧 Phase 5 を破棄）。
- **アーキテクチャ**：キャラ属性は**従来どおり `characters/{id}/*.json` に外出し**（CLAUDE §4.5/§5.1「明示的>暗黙的・ハードコード禁止」を維持）。
  - 利点：(a) 工学的退化なし、(b) コードは特定キャラに非依存のまま、(c) 万一トリミが刺さらなくても**コード改変なしで作り直せる**（賭けの可逆性）。
- **命名の整理**：**`ENE` = アプリ/プロジェクトのコードネーム**、**`魚川トリミ` = キャラクター名**。
  既存文書・コードの `characterId` は当面 `"ene"` のまま（識別子の churn を避ける）。

### 1.2 データ影響：実質なし

- 記憶は単一 `characterId` 配下に集約済み。複数キャラ汎用性のための**横展開テストは不要**になる（簡素化）。
- ⚠️ **本改訂で触らない作業（別途・創作タスク）**：`identity.json` の `name` を `"魚川トリミ"` へ、
  `fewshot.json`・挨拶台詞の `"ENE"` 自称を新名へ改める作業と、§2 の人生記憶 canon 執筆。
  これらは**人格テキストの創作**であり、本改訂（スキーマ＋方針）とは分離する。

## 2. 決定2：キャラ自身の人生記憶（authored canon）

「そういえば昔、私も…」のように、キャラ自身の過去を会話に織り交ぜる。**作家が書いた正典(canon)**を固定し、
Claude にその場で生い立ちを創作させない（継続性崩壊＝柱2「一貫した人格」違反を防ぐ）。

### 2.1 provenance（出所）フィールド

```typescript
// src/shared/types/memory.ts（EpisodicMemory に追加・optional）
provenance?: 'user' | 'self';   // ★新 欠落時は 'user' 扱い。
                                 //   'self' = キャラ自身の人生記憶(canon)
```

- `'user'`：ユーザーとの関係の中で生まれた記憶（既存の episodic）。読み書き対象・忘却対象。
- `'self'`：キャラ自身の人生（canon）。**読み取り専用・忘却対象外・supersede 対象外**。

### 2.2 canon の置き場所（キャラ資産・配布物）

人生記憶は**ユーザーデータではなくキャラプロファイルの一部**（出荷時に同梱・作家が執筆）。
よって `data/memory/`（ユーザー領域）ではなく `characters/{id}/` 配下に置く。

```
characters/{id}/
├── identity.json
├── background.json          ← 静的な背景設定(出身・家族・趣味)。人生記憶の“事実の素”
├── knowledge_domains.json
├── fewshot.json
├── life-memory.json         ← ★新 人生エピソードの canon 配列(provenance:self)
└── portrait.png
```

- `life-memory.json` は `EpisodicMemory`（`provenance:'self'`・`importance` 高め・`valence` 付き）の**配列**。
- 平文JSON（§6.1 可読性・可搬性）。**読み取り専用**でユーザー領域へはコピーしない（=忘却・改変されない不変の canon）。
- 想起時に**user episodic と統合プールにマージ**して横断検索（処理は task_15/16）。

### 2.3 作話防止・整合性（柱2 / 4層防御の前例）

- **canon 外の伝記的事実を新規創作しない**ようプロンプトで制約（「生い立ちは与えられた life-memory の範囲でのみ語る」）。
  AI自称防止の4層防御と同じ「出力ガード」の系譜。
- **`knowledge_domains.json` と矛盾しない**：少女が `none` の領域（パチンコ・投資・運転 等）を人生記憶に含めない。
  canon 執筆時に知識濃淡表と突き合わせる運用とする。

## 3. 決定3：心（感情価バイアス想起・**導出**方式）

ユーザーとの会話が、キャラが**どの記憶を想起しやすいか**に微細なバイアスを与える（気分一致記憶 / mood-congruent recall）。
人格・性格は固定だが、**1〜2年の蓄積で唯一無二の会話体験**を作る。

### 3.0 方針転換：永続スカラー方式は採らない

検討の結果、「心情を −100〜+100 のスカラーで保存し、日次で±1・週次で0へ回帰」させる案は**採用しない**。理由：

1. **§5.3 整合**：状態（感情パラメータ）を**貯めない**方針を保てる。
2. **部品最小**：スカラー案は「日次分類ジョブ＋週次回帰ジョブ＋保存状態」を要するが、**それでも各記憶の valence は別途必要**。
   下記の導出方式は **valence フィールド＋想起時の数行の算術**だけで、**スカラー案より部品が少ない**。
3. **加害回避**：脆弱なユーザーへ無制限に同調して暗転し続けるリスク（倫理の一線 §4）を、**非対称な復元力**で抑える。

> **心情は「貯めず、記憶から読み出す」。** 永続性は心情パラメータではなく**記憶コーパスの蓄積**が担う
> （負の出来事が増えれば負 valence の記憶が増える＝自然に永続）。減衰は**直近重み付け**がそのまま担う（沈黙すれば0へ）。

### 3.1 valence（各記憶の感情価・中立観察）

```typescript
// EpisodicMemory に追加・optional
valence?: number;   // ★新 -2..+2 の整数。欠落時は 0(中立)扱い。
                    //   出来事の感情的トーン(うれしい=正/つらい=負)。
                    //   importance(=忘却の重み)とは別軸。感情“管理”ではない。
```

- **抽出時に付与**。抽出は**中立観察者**のまま（「この出来事はユーザーにとって負だった」は観察であり口調ではない）。
- 意味は**出来事のトーン**に限る（v1）。「関係の温度（通じ合った/突き放された）」は別軸で v1 非対象。
  （理由：愚痴＝話題は負だが関係は深まる、を取り違えないため。混ぜると安っぽくなる。）

### 3.2 心情の導出（保存しない・毎想起で再計算）

```
mood_global = Σ wᵢ·valenceᵢ / Σ wᵢ        （直近の episodic を対象）
wᵢ = exp(-Δdaysᵢ / τ)
  τ_pos ≈ 14日（正の記憶）, τ_neg ≈ 7日（負の記憶）  ← ★非対称：負は速く減衰＝復元力
```

- **状態ファイルなし**。会話のたびに手元の記憶から算術で出す（LLM 呼び出し不要・ローカル）。
- 沈黙すれば直近重みが下がり**自然に 0（中立）へ**（＝「しばらく話さなければ戻る」が無償で得られる）。
- 時定数 τ で「数週間ひきずる」を調律（1パラメータ）。

### 3.3 想起バイアス（RRF に1項足すだけ）

task_15 の `MemoryRetriever`（ベクトル＋語彙＋entity を RRF 合流）に、**バイアス項**を加える：

```
finalScore(m) = RRF(m) + λ · clampedMood · valence(m)
clampedMood   = max(mood_global, MOOD_FLOOR)   // 下げ過ぎ防止＝“デレの床”
```

- `λ` は**小さく**：基底スコアが拮抗した時のみ順位が入れ替わる（＝「微細」）。
- 最終選択は **softmax サンプリング**（決定論的に「負なら必ず喧嘩を出す」を避け、揺らぎを与える）。
- **「微細」は単一ターンの性質ではなく、多数の想起にわたる統計的性質**（小さな候補集合では1ターンは二値になりうる）。

### 3.4 相手別バイアス（別システムではない・後付け）

retriever はクエリの entity（例「田中」）を既に知っている。**同じ式に entity フィルタをかけるだけ**：

```
topicMood_E = 上式を  m ∈ (entity E を含む記憶)  に限定して算出
```

- **v1 は global のみ**で出す（最小）。相手別は同じ関数を範囲限定で呼ぶ**数行の追加**（アーキテクチャ変更ゼロ）。
- 複雑さは「global か相手別か」の二択ではなく、`λ` の調整に縮む。

### 3.5 出力は荒らさない（“壊れたキャラ”回避・Packard 2019）

- mood は**想起の重みだけ**を動かす。応答トーンの制約・**デレ（ケア）は人格層で常時**かかる（−側でも暴言は出ない）。
- mood の**数値はユーザーに見せない**（導出値・真実の源は記憶コーパス）。デバッグで見たい場合のみ**読み取り専用キャッシュ**として書ける（§6.1 整合・“貯まる好感度メーター”にはしない）。

### 3.6 持たないもの（§5.3 の確認）

- ❌ 保存される心情スカラー / 好感度 / 怒り度。 ❌ 日次・週次の更新ジョブ。
- `importance`（忘却の重み）・`valence`（想起の色）は**感情の数値管理ではない**。

## 7. 設計判断の記録（実装時に implementation-notes へ N-16-x として転記）

- N: 製品は単一固定キャラ（魚川トリミ）。**アーキテクチャの JSON 外出しは維持**（ハードコード禁止・可逆性確保）。旧 Phase 5（多キャラPF）破棄。
- N: `EpisodicMemory` に `provenance?`（user/self）・`valence?`（-2..+2）を追加。全 optional・後方互換。
- N: 人生記憶 canon は**キャラ資産** `characters/{id}/life-memory.json`（読取専用・忘却外）。ユーザー領域へコピーしない。
- N: 心は**永続スカラーを持たず記憶から導出**（mood_global・直近重み付き平均）。日次/週次ジョブ不要。スカラー案より部品が少ない。
- N: 減衰は**非対称**（τ_neg < τ_pos）＋ `MOOD_FLOOR`（デレの床）で**暗転ロックを回避**（脆弱ユーザー対策・倫理の一線）。
- N: 想起バイアスは RRF に `λ·clampedMood·valence(m)` を加算＋softmax サンプリング。「微細」は統計的性質。
- N: 相手別は同式に entity フィルタ（別システムではない）。v1=global、相手別は後付け。
- N: mood は出力トーンを荒らさない（想起の重みのみ）・数値はユーザー非表示（§5.3/§6.1 整合）。

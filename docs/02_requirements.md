## 1. 全体要件

### 1.1 プロダクト名
Desktop Character Agent(コードネーム:ENE)

### 1.2 対象プラットフォーム
- Windows 10 / 11

### 1.3 配布形態
- Electronによる単一パッケージ(exe)
- **インストール不要のポータブル形式**(exeと同じディレクトリに全データを保持)

### 1.4 リリース段階と実装状況

「コア体験を成立させる最小構成」(0.1)から段階的に育てる。各段階の背骨と
現在の実装状況は以下のとおり(段階の全体像・ロードマップは `01_vision.md` §6/§9 を参照)。

#### ✅ 0.1「存在の核」(実装済み)
- キャラクターの透過表示(PNG)
- テキスト入力による会話(Claude API)
- Character Knowledge Profile(キャラ人格・知識管理)
- Knowledge Router(知識ドメイン動的判定)
- 3層記憶システム(短期・中期・長期、タグ検索ベース)
- ホワイトリスト方式のOS操作(メモ帳・ブラウザ・フォルダ)
- アプリ動作ログ(個人情報を含まないメタ情報のみ)
- 誕生日判定機能(任意属性)

#### ✅ 0.2「存在感」(実装済み・task_13)
- アニメ基盤(状態機械 idle/thinking/talking＋emotion 表情差分・口パク)→ §2.1 F-ANIM
- クリック音等の UI 音(Web Audio 合成)

#### ✅ 0.3「声と耳・記憶の深化」(実装済み・task_14〜17)
- 双方向ローカル音声(STT=Whisper / TTS=AivisSpeech / VAD=Silero v4・barge-in)→ §2.14 F-VOICE
- 記憶想起エンジン(ベクトル＋語彙＋entity・RRF)と非破壊更新 → §2.9
- 単一固定キャラ・人生記憶 canon・心(感情価バイアス想起)→ §2.6/§2.9
- プロンプトキャッシュによる入力トークン最適化

#### ⬜ 今後(1.0 以降・最適化フェーズ)
- 人間的忘却(月次/年次再要約・低重要度削除)→ §2.9 忘却機構
- 高度なOS操作(ファイル操作・クリップボード等)
- C1 ストリーミング音声(レイテンシ最適化)・振幅ドリブンのリップシンク
- Live2D / three-vrm による本格モーション
- 設定画面の拡充
- 既知のブラッシュアップ項目 → `docs/optimization-backlog.md`

#### 🚫 恒久的に持たないもの(方針として不採用)
- 複数キャラクター切替UI・キャラ配布(固定キャラ方針・2026-06 方針転換)
- 保存される数値感情/好感度スカラー(心は記憶からの導出のみ)
- 専用モデル(LoRA 等)

## 2. 機能要件

### 2.1 キャラクター表示

| ID | 要件 |
|----|------|
| F-DISP-01 | 透過ウィンドウにPNGキャラクターを表示する |
| F-DISP-02 | ウィンドウは常に最前面に固定される |
| F-DISP-03 | キャラクター画像をドラッグして移動可能 |
| F-DISP-04 | ウィンドウ位置は終了時に保存され、次回起動時に復元される |
| F-DISP-05 | 初回起動時は画面右下にデフォルト配置される |
| F-DISP-06 | アニメーション(差分2〜3枚の切り替え)は任意実装とする |
| F-DISP-07 | キャラクター上で右クリックすると、コンテキストメニューを表示する |

#### アニメ基盤(F-ANIM・MVP 0.2・task_13 で実装)

| ID | 要件 |
|----|------|
| F-ANIM-01 | キャラ表示は状態機械(activity: idle/thinking/talking, emotion, pose)で駆動する |
| F-ANIM-02 | スプライト・状態→画像の対応・フレーム間隔は `characters/{id}/animation.json` で定義(コードにハードコードしない) |
| F-ANIM-03 | idle 中は呼吸の微動(CSS)を再生し、長時間 idle で稀に寝そべり(sofa)へ遷移する(まばたきは 0.2 では未実装) |
| F-ANIM-04 | 応答待ち(thinking)中は「考える」表示を出し、遅延をキャラ化する |
| F-ANIM-05 | 応答表示(talking)中は口パク(開閉交互)を再生。0.2 は時間ベース・メッセージ長に比例、0.3 で音声振幅ドリブンに差し替える |
| F-ANIM-06 | 応答に emotion 離散ラベルを含め対応表情へ切替。欠落・許可外は `neutral` にフォールバック |
| F-ANIM-07 | emotion は1ターン揮発ラベル。好感度・不機嫌度などの数値状態は持たない(§5.3) |
| F-ANIM-08 | クリックスルー判定(`isOpaqueAt`)は現在表示中フレームの alpha で行う |
| F-ANIM-09 | 視覚レンダラは `CharacterState` インターフェースで差し替え可能(将来 Live2D/VRM) |
| F-ANIM-10 | クリック音等の UI 音は Web Audio で合成する(外部音源を同梱しない) |
| F-ANIM-11 | アニメ定義が無い・不正な場合は単一 portrait 表示にフォールバックする(後方互換) |
| F-ANIM-12 | フレーム間隔・呼吸/口パク間隔等の数値は定数として一元管理する |

### 2.2 テキスト入力・出力

| ID | 要件 |
|----|------|
| F-IO-01 | キャラクター付近にテキスト入力欄を設置する |
| F-IO-02 | デフォルトでは入力欄は非表示(キャラのみ表示) |
| F-IO-03 | キャラの短いクリック(動かさず素早く離す)で入力欄を展開する |
| F-IO-04 | ESCキーで入力欄を閉じる |
| F-IO-05 | ユーザー入力テキストはClaude APIに送信される |
| F-IO-06 | 応答はキャラクター付近に吹き出しとして表示される |
| F-IO-07 | 入力テキストはUTF-8文字列として扱う |

#### 吹き出しのライフサイクル

| ID | 要件 |
|----|------|
| F-BUBBLE-01 | 吹き出しは ENE の応答時に表示される |
| F-BUBBLE-02 | 吹き出しは横幅固定(キャラ幅相当、約240px)で縦方向に自動拡張する |
| F-BUBBLE-03 | 吹き出しの最大高さは 400px とし、超過時は内部スクロールを許可する |
| F-BUBBLE-04 | 表示から 30 秒経過で自動消滅する |
| F-BUBBLE-05 | ユーザーが新しい入力を送信した時点で、古い吹き出しは即座に消える |
| F-BUBBLE-06 | 吹き出しをクリックすると即座に閉じる |
| F-BUBBLE-07 | ESCキー押下でも吹き出しは閉じる |
| F-BUBBLE-08 | 数値(30秒・400px等)は定数として一元管理する |

### 2.3 透過ウィンドウとマウス操作

#### 透明領域のクリックスルー

| ID | 要件 |
|----|------|
| F-WIN-01 | キャラの不透明ピクセル上ではマウスイベントを受け取る(キャラと対話可能) |
| F-WIN-02 | キャラの周りの透明領域では、マウスイベントを下のデスクトップに通す(クリックスルー) |
| F-WIN-03 | 透明領域上のクリック・右クリックは、デスクトップアイコン等の操作として処理される |
| F-WIN-04 | この挙動は Electron の `setIgnoreMouseEvents(true, { forward: true })` で実装する |

#### マウス操作の判別(クリック / ドラッグ / 長押し)

キャラクター上のマウス操作は、移動距離と時間で以下の3種類に判別する。

| ID | 要件 |
|----|------|
| F-MOUSE-01 | mousedown から mousemove までの移動距離が 5px 以上 → ドラッグと判定し、ウィンドウ移動を開始する |
| F-MOUSE-02 | 移動 5px 未満、押下時間 500ms 未満で mouseup → クリックと判定し、入力欄を展開する |
| F-MOUSE-03 | 移動 5px 未満、押下時間 500ms 以上で mouseup → 長押し扱いで何もしない(誤操作回避) |
| F-MOUSE-04 | 一度ドラッグと判定した後は、その mousedown 期間中はクリック判定にならない |
| F-MOUSE-05 | 数値(5px / 500ms)は将来的な調整可能性を考慮し、定数として一元管理する |

### 2.4 アプリ終了とタスクトレイ

| ID | 要件 |
|----|------|
| F-EXIT-01 | タスクトレイにアイコンを常駐表示する |
| F-EXIT-02 | タスクトレイアイコンのシングルクリックでウィンドウ表示/非表示を切り替える |
| F-EXIT-03 | タスクトレイの右クリックメニューから「ENE を終了」できる |
| F-EXIT-04 | キャラクター上の右クリックメニューからもアプリを終了できる |
| F-EXIT-05 | ウィンドウの「×」相当のボタンは持たない(フレームレス設計のため) |
| F-EXIT-06 | 終了メニュー項目の文言はキャラ口調で表現する(例:「じゃあね...」) |

#### 初回起動時のユーザーガイド(R7)

| ID | 要件 |
|----|------|
| F-FIRST-01 | 初回起動時は ENE が自己紹介と操作方法をキャラ口調で説明する |
| F-FIRST-02 | 操作説明用のチュートリアルダイアログは使用しない(キャラ口調で代替) |
| F-FIRST-03 | 初回起動完了フラグを `active-character.json` に最小状態として記録する |
| F-FIRST-04 | 2回目以降の起動では通常の挨拶(「おかえり」等)を表示する |

### 2.5 会話処理

| ID | 要件 |
|----|------|
| F-CONV-01 | Claude APIへの送信時、キャラクター人格情報を含むプロンプトを構築する |
| F-CONV-02 | プロンプトには知識ドメインに応じた振る舞い指示を含む |
| F-CONV-03 | プロンプトには関連する記憶コンテキストを含む |
| F-CONV-04 | 出力形式はJSON固定とする |
| F-CONV-05 | 応答はchat型とos_command型の2種類を扱う |
| F-CONV-06 | JSON 出力の安定化は system プロンプトの強い指示＋ロバストパーサ(F-CONV-08/09)で行う。**現行 Claude 4.x は assistant メッセージ Prefill 非対応のため Prefill は使わない**(N-09-7) |
| F-CONV-07 | temperature は 0.7 を採用する(キャラの自然さと JSON 安定性のバランス) |
| F-CONV-08 | 応答パース時にコードフェンス(```json ... ```)があれば除去する |
| F-CONV-09 | 応答の前後にテキスト混入があっても `{` 〜 `}` の範囲を抽出してパースする |
| F-CONV-10 | パース失敗・型不正の場合、キャラ口調のフォールバック応答を返す |
| F-CONV-11 | プロンプトに `identity.json` の `neverCallsSelf` を明示し、キャラがAI自称しないよう指示する |
| F-CONV-12 | 応答取得後、chat メッセージ内に `neverCallsSelf` の語が含まれていないか検知する |
| F-CONV-13 | AI自称が検知された場合、強化プロンプトで再生成を1回試みる |
| F-CONV-14 | 再生成でも検知された場合、キャラ口調のフォールバック応答を返す |
| F-CONV-15 | AI自称検知のロジックは `identity.json` を参照する(コードへのハードコード禁止) |

#### 応答型:chat
キャラクターの返答を吹き出しに表示する。

#### 応答型:os_command
OS Integration Layerにコマンド実行を委譲し、付随するキャラ応答も表示する。

### 2.6 キャラクター人格管理

| ID | 要件 |
|----|------|
| F-CHAR-01 | キャラクター属性は外部JSON設定ファイルで管理する |
| F-CHAR-02 | キャラクター定義は4ファイル構成(identity / background / knowledge_domains / fewshot) |
| F-CHAR-03 | キャラクターは「歳を取らない」(時間経過で属性が変化しない) |
| F-CHAR-04 | 年齢は具体的な数字ではなく抽象表現(例:「少女」)で扱う |
| F-CHAR-05 | キャラクターは自分を「AI」「アシスタント」「モデル」等と呼ばない |
| F-CHAR-06 | 知識ドメインは5段階(high / medium / low / none / refuse)で管理する |
| F-CHAR-07 | 各ドメインに対応するFew-shot応答例を保持する |
| F-CHAR-08 | **一人の固定キャラ(魚川トリミ)**とする。キャラ依存値は JSON 外出しを維持するが、入れ替えUI・多キャラ対応は持たない(2026-06 方針転換) |
| F-CHAR-09 | キャラは自身の人生記憶(canon)を `characters/{id}/life-memory.json` に持てる(任意・provenance:self・読取専用)。想起プールに統合し会話に織り交ぜる(実装済み・task_16) |

### 2.7 知識ドメイン判定(Knowledge Router)

| ID | 要件 |
|----|------|
| F-ROUTE-01 | ユーザー入力のトピックを判定する |
| F-ROUTE-02 | 判定結果をknowledge_domainsと照合し、該当ドメインを特定する |
| F-ROUTE-03 | 該当ドメインの振る舞い指示とFew-shotキーを返す |
| F-ROUTE-04 | 判定不能・タイムアウト・API失敗時はfallbackドメインを使用する |
| F-ROUTE-05 | Knowledge Routerの失敗は本会話の進行を妨げない(ベストエフォート方式) |
| F-ROUTE-06 | 直近の判定結果を一定数キャッシュする(同一入力時の高速化) |

### 2.8 誕生日機能(任意)

| ID | 要件 |
|----|------|
| F-BDAY-01 | キャラクターは任意で誕生日(月日)を持つことができる |
| F-BDAY-02 | 誕生日当日は、キャラ固有の特別反応(`celebrated`)を返せる |
| F-BDAY-03 | 当日にユーザーが触れずに過ぎた場合、翌日以降に「忘れられた」反応(`forgotten`)を返せる |
| F-BDAY-04 | 反応の具体的な文言は Few-shot 例として実装する(感情パラメータの数値管理は行わない) |
| F-BDAY-05 | 「祝われた / 祝われていない」の二値の事実のみ最小状態として記録する(`active-character.json` に保存) |
| F-BDAY-06 | 誕生日を持たないキャラクターも許容する(必須属性ではない) |
| F-BDAY-07 | キャラクター自身は加齢しない(誕生日を迎えても年齢表現は変わらない) |

### 2.9 記憶システム

#### 短期記憶

| ID | 要件 |
|----|------|
| F-MEM-S-01 | 直近の会話内容(セッション内)を保持する |
| F-MEM-S-02 | 保持件数の上限を設ける(超過時は古いものから削除) |
| F-MEM-S-03 | LLMのコンテキストに直接渡される |
| F-MEM-S-04 | 各エントリは「抽出済みフラグ(`extracted`)」を持つ |
| F-MEM-S-05 | 保持件数超過時、削除前に未抽出エントリを中期記憶へ抽出する |
| F-MEM-S-06 | アプリ終了時、未抽出エントリを中期記憶へ抽出する |
| F-MEM-S-07 | 抽出処理は `extracted: false` のエントリのみを対象とする(重複防止) |

#### 中期記憶(Episodic)

| ID | 要件 |
|----|------|
| F-MEM-E-01 | 会話から抽出された出来事・事実を保存する |
| F-MEM-E-02 | 保存項目:日付・トピック・要約・タグ・重要度・カテゴリ |
| F-MEM-E-03 | JSON形式でファイルごとに保存する |
| F-MEM-E-04 | カテゴリ別・年別の階層構造で保存される |
| F-MEM-E-05 | 全会話ログは保存しない(要約のみ) |
| F-MEM-E-06 | 重要度は1〜5の整数で必須項目とする(将来の忘却機構で参照されるため) |

#### 長期記憶(Semantic)

| ID | 要件 |
|----|------|
| F-MEM-L-01 | ユーザーの性格・好み・価値観・人生の事実を保持する |
| F-MEM-L-02 | 変化の少ない情報のみを保存する |
| F-MEM-L-03 | 常にプロンプトの一部として読み込まれる |
| F-MEM-L-04 | 単一のJSONファイルで管理される |

#### 記憶の保存・読込フロー

| ID | 要件 |
|----|------|
| F-MEM-F-01 | 会話終了時、LLMが会話内容から記憶抽出を行う |
| F-MEM-F-02 | 抽出結果に応じて中期・長期記憶を更新する |
| F-MEM-F-03 | 会話開始時、短期・長期記憶を読み込む |
| F-MEM-F-04 | 会話中、テーマに基づき関連する中期記憶を検索・読込する |
| F-MEM-F-05 | 検索結果の上限は5〜10件とする |

#### 検索仕様

| 検索方法 | 対応 |
|----------|---------|
| タグ検索 | ✅ |
| カテゴリ検索 | ✅ |
| 重要度フィルタ | ✅ |
| 日付フィルタ | ✅ |
| ベクトル検索(意味検索) | ✅ (0.3・task_15・ローカル埋め込み ruri) |
| 語彙＋entity 逆引き・RRF 合流 | ✅ (0.3・task_15) |
| LLM再ランキング | ❌ (将来拡張) |

#### 忘却機構(将来拡張・MVP対象外)

ビジョン§3 柱1「人間らしい忘却」を実現する機構。MVPでは実装しないが、
**MVPの設計はこの機構を後から追加できる構造とする**。

| ID | 要件 |
|----|------|
| F-MEM-X-01 | (将来)月次・年次・5年単位で中期記憶を再要約する |
| F-MEM-X-02 | (将来)重要度の低い記憶から優先的に削除する |
| F-MEM-X-03 | (将来)再要約された記憶は元の詳細を削除する(容量削減・人間らしい忘却) |
| F-MEM-X-04 | (将来)ユーザーが「この記憶を忘れて」と指示できる |

MVPでは Episodic Memory に `importance` を必須化することで、
将来の忘却機構の入口を確保する(F-MEM-E-06)。

#### 人生記憶・心(感情価バイアス想起)(実装済み・0.3・task_15/16)

ビジョン§3 柱1/柱2、哲学§6 に基づく。

| ID | 要件 |
|----|------|
| F-MEM-H-01 | キャラ自身の人生記憶(canon)を想起プールに統合し、会話に織り交ぜる(provenance:self・読取専用) |
| F-MEM-H-02 | 各記憶に valence(感情価・中立観察)を付与する(-2〜+2・任意) |
| F-MEM-H-03 | 想起時、直近の会話から**心情を導出**し、想起にゆるやかなバイアスをかける(保存スカラーは持たない) |
| F-MEM-H-04 | 心情バイアスは想起の重みのみに作用し、応答トーン・キャラのデレを荒らさない |
| F-MEM-H-05 | 心情の数値はユーザーに表示しない(導出値・好感度メーター化を避ける) |
| F-MEM-H-06 | 親しさ(接触の事実=日数・会話日数・回数から導出・単調非減少)に応じて記憶の開示段階(disclosureLevel)を制御する。保存される好感度スカラーは持たない |

データモデル・アルゴリズムは `docs/03_design.md` §3.3、処理は `tasks/task_16_*.md`。

### 2.10 OS操作

| ID | 要件 |
|----|------|
| F-OS-01 | OS操作はホワイトリスト方式で厳格に制限する |
| F-OS-02 | MVPで対応する action は3種類のみ:`open_notepad` / `open_browser` / `open_folder` |
| F-OS-03 | `open_browser` の target URL は http/https のみ許可する |
| F-OS-04 | `open_folder` の target パスはユーザーホームディレクトリ配下のみ許可する |
| F-OS-05 | 上記以外の action ・許可されないURL/パスは即座に拒否する |
| F-OS-06 | ブラウザ・フォルダ起動には Electron `shell` API を使用する(シェル経由を避ける) |
| F-OS-07 | 実行結果はキャラ応答として表示される |
| F-OS-08 | コマンド実行失敗時は、キャラ口調でエラーを伝える |

### 2.11 アプリケーション動作ログ

ユーザーの会話内容を逐語的に記録する「会話ログ」機能は **MVP では実装しない**。
過去の会話を辿る用途は、中期記憶(Episodic Memory)の `summary` フィールドが
果たす(F-MEM-E-02 / F-MEM-E-05)。これはビジョン§3 柱1
「全会話ログを保持しない・人間らしい忘却」と整合する。

代わりに、アプリの**動作・障害解析のためのデバッグログ**のみ実装する。

| ID | 要件 |
|----|------|
| F-LOG-01 | アプリの起動・終了・エラーを `electron-log` で記録する |
| F-LOG-02 | ログレベルは error / warn / info / debug の4段階を使用する |
| F-LOG-03 | ログには**会話内容(ユーザー入力・AI応答の全文)を含めない** |
| F-LOG-04 | ログには**プロンプト全文・記憶コンテキスト等の個人情報を含めない** |
| F-LOG-05 | デバッグログに記録するのはメタ情報のみ(API応答時間・Router判定結果・エラー種別等) |
| F-LOG-06 | ログファイルはユーザーが任意に削除可能 |
| F-LOG-07 | ログファイルは平文(可読性のため。ただし個人情報を含まない構造) |

### 2.12 APIキー管理

| F-KEY-01 | APIキーは Electron safeStorage で暗号化して保存する |

### 2.14 音声入出力(双方向ローカル音声・MVP 0.3「声と耳」・task_17)

処理はすべてローカル。脳(Claude)へ送るのは**テキストのみ**(音声は端末外に出さない・§3.4 プライバシー / ビジョン§4.2 維持)。
詳細設計は `docs/archive/design-revision-voice.md`、実装判断は `docs/implementation-notes.md` N-17-x を参照。

| ID | 要件 |
|----|------|
| F-VOICE-01 | 音声合成(TTS)は `TtsEngine` インターフェースで差し替え可能とする(§4.4 疎結合)。MVP 0.3 実装は AivisSpeech(VOICEVOX互換のローカル HTTP API・既定 `127.0.0.1:10101`)。音声を外部クラウドへ送らない(ローカル合成) |
| F-VOICE-02 | emotion ラベル→スタイルID/合成パラメータの対応は `characters/{id}/voice.json` に外出しする(コードにハードコードしない・§4.5)。`neutral` を必須フォールバックとする。`pitchScale` は音質劣化のため使わない(声質はモデル選択＋スタイルで決める) |
| F-VOICE-03 | 応答は読み上げ用ひらがな `reading`(任意)を持てる。TTS は `reading` があればそれを、無ければ表示文 `message` を読む(漢字の誤読対策) |
| F-VOICE-04 | 確定応答を文単位で合成し、文ごとに音声チャンク(WAV)を renderer へ送って逐次再生する(吹き出しは即時表示・音声は後追いで届く) |
| F-VOICE-05 | 音声認識(STT)は完全ローカル(main プロセス・`onnxruntime-node`・`whisper-large-v3-turbo`)。録音音声は外部送信せず STT 入力にのみ使う。脳(Claude)へ送るのは確定テキストのみ |
| F-VOICE-06 | マイク取得は renderer の `getUserMedia`(16kHz mono)。STT モデルは別ダウンロードで `data/models/` 配下へ配置する(コア配布物 <100MB を維持・実行時に外部からモデルを取らない) |
| F-VOICE-07 | ハンズフリー会話では Silero VAD(**v4**・main・`onnxruntime-node`・同梱)で発話区間を検出し、無音継続(既定 700ms)でターン終了とみなして STT→既存 `sendMessage` 経路へ流す |
| F-VOICE-08 | barge-in:ENE 発話中(実音声の再生中)にユーザー発話を検知したら TTS 再生を即停止する。エコー回り込みでの誤発火対策に `echoCancellation` を有効化する |
| F-VOICE-09 | マイク入力方式(push-to-talk / hands-free)を設定可能とし、`data/config/app-settings.json`(平文JSON)に永続化する。キャラ右クリックメニューから切替できる |
| F-VOICE-10 | 同梱・配布する音声モデルは寛容ライセンス(声優の無断クローン等を含まないもの)に限定する。クレジット必須の声は、クレジット文言を about/クレジット画面に常時表示する |
| F-VOICE-11 | 音声機能は best-effort:エンジン未起動・モデル未配置・合成失敗時でもテキスト会話は成立する(音声無効でフォールバックし、会話をブロックしない) |

## 4. アーキテクチャ要件(レイヤー構成)

本プロダクトは以下の5レイヤーで構成される。各レイヤーは疎結合とし、
インターフェース経由でのみ通信する。

### 4.1 全体構成

```
UI Layer(Electron Renderer)
   ↓ テキスト
Knowledge Router(トピック判定)
   ↓ ドメイン情報
   ┌─────────────────┬──────────────────┐
   ↓                  ↓                  ↓
Character Layer    Memory Layer    (両者を統合)
(人格・知識)        (記憶システム)
   └─────────────────┴──────────────────┘
                      ↓ 構築済みプロンプト
Conversation Layer(Claude API)
   ↓ JSON応答
   ├─ Memory Layer(記憶の保存・更新)
   └─ OS Integration Layer(コマンド実行)
```

### 4.2 各レイヤーの役割

| レイヤー | 主な責務 |
|----------|----------|
| UI Layer | キャラ表示、入力受付、吹き出し表示 |
| Knowledge Router | ユーザー入力のトピック判定、ドメイン照合 |
| Character Layer | キャラ人格・背景・知識プロファイルの管理 |
| Memory Layer | 3層記憶の保存・検索・更新 |
| Conversation Layer | Claude API呼出、プロンプト構築、応答解析 |
| OS Integration Layer | ホワイトリスト方式のコマンド実行 |

### 4.3 レイヤー間インターフェース要件

| 要件 |
|------|
| 全インターフェースは型定義された契約として実装する |
| 上位レイヤーは下位レイヤーの実装詳細に依存しない |
| 任意のレイヤー内部実装を差し替えても、他レイヤーは変更不要であること |

具体的な型定義・関数シグネチャは `03_design.md` 第3章を参照のこと。

## 5. データ要件

### 5.1 キャラクタープロファイル(配布物)

キャラクター1体につき以下のファイルで構成する。

| ファイル | 内容 |
|----------|------|
| identity.json | 人格コア(名前・口調・性格・誕生日等) |
| background.json | 背景設定(出身・家庭・趣味等) |
| knowledge_domains.json | 知識ドメイン定義(5段階) |
| fewshot.json | ドメイン別の応答例集 |
| portrait.png | キャラクター画像 |

### 5.2 ユーザーデータ(実行時生成)

部分暗号化方式に従い、データは2箇所に分散して保存される。

#### ポータブルデータ(exeと同じディレクトリの `data/` 配下)
全て平文JSON。ユーザーが直接読める。exeごとコピーで別PCに持ち運び可能。

| ディレクトリ・ファイル | 内容 |
|----------------------|------|
| data/memory/{characterId}/short-term.json | 短期記憶(セッション内、キャラ別) |
| data/memory/{characterId}/semantic.json | 長期記憶(性格・好み等、キャラ別) |
| data/memory/{characterId}/episodic/{year}/{category}/ | 中期記憶(キャラ別) |
| data/config/active-character.json | 現在使用中キャラと最小状態 |
| data/config/window-position.json | ウィンドウ位置 |
| data/logs/ | アプリ動作ログ(個人情報を含まないメタ情報のみ) |

#### マシン固定データ(OS のユーザー領域)
暗号化されており、別PCに持ち運んでも復号できない。再入力が必要。

| ディレクトリ・ファイル | 内容 |
|----------------------|------|
| %APPDATA%/ene-desktop/api-key.enc | 暗号化APIキー |

### 5.3 データの所有権と管理

| 要件 |
|------|
| ユーザーは `data/` ディレクトリを自由にバックアップ・削除できる |
| ユーザーはexeディレクトリごと別PCにコピーして継続利用できる(APIキーのみ再入力) |
| 完全なアンインストールには `data/` と `%APPDATA%/ene-desktop/` の両方の削除が必要 |
| アプリ起動時、`%APPDATA%/ene-desktop/` にAPIキーが無ければセットアップダイアログを表示する |

## 7. 受入基準(MVP完成条件)

MVP完成時、以下がすべて成立していること(ビジョン§7「成功基準」より)。

| # | 基準 | 判定方法 |
|---|------|---------|
| 1 | ENEがデスクトップ上に常駐し、ドラッグで移動できる | 目視・操作確認 |
| 2 | テキストで話しかけると、ENEのキャラとして応答が返ってくる | 目視 |
| 3 | ENEに「メモ帳を開いて」と頼むと、開いてくれる | 動作確認 |
| 4 | ENEに「パチンコの新台教えて」と聞くと、自然に「知らない」と返す | 動作確認 |
| 5 | 前回の会話を覚えていて、関連する話題で思い出してくれる | 動作確認(複数セッション) |
| 6 | APIキーは暗号化されて保存され、外部送信はClaude APIへの会話のみ | コード確認 + 通信監視 |
| 7 | 配布サイズ100MB以下、常駐CPU 3%以下を達成 | 計測 |
| 8 | ユーザーが「AIっぽくない、ENEと話している」と感じる | **手動確認(人間判定)** |

特に **8番** は本プロダクトの本質であり、他の基準を満たしても
8番を満たさない場合は失敗と判定する。

8番の具体的な手動確認プロトコルは **ビジョン§7「成功基準8の手動確認プロトコル」**
を参照のこと。`tests/acceptance/manual-check.md` 等で管理する。

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| Character Profile | キャラクターの人格・背景・知識を定義したJSONファイル群 |
| Knowledge Domain | キャラクターが知っている領域の分類(high/medium/low/none/refuse) |
| Knowledge Router | ユーザー入力からドメインを判定するレイヤー |
| Short-term Memory | 直近の会話を保持する短期記憶 |
| Episodic Memory | 出来事・事実を要約して保存する中期記憶 |
| Semantic Memory | 性格・好み・価値観を保持する長期記憶 |
| Few-shot | LLMに振る舞いを学習させるための応答例 |
| Fallback | 判定不能・失敗時のデフォルト動作 |
| ポータブル | exeディレクトリにデータを集約し、可搬性を確保する方式 |

---

*本要件定義書は、ビジョンと実装の橋渡しである。*
*「何を作るか」を厳密に定義し、「どう作るか」は設計書に委ねる。*

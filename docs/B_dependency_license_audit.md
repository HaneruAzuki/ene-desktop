## 1. 用語と判定基準

| 判定 | 意味 |
|------|------|
| ◎ 無条件 | 売上・用途を問わず無償。寛容ライセンス(MIT/Apache/BSD)。義務はライセンス表記程度 |
| ○ 条件あり | 無償だが、クレジット義務・コピーレフト・禁止用途などの条件がある |
| △ 売上連動 | 一定売上を超えると有償化する(=純粋性を破る) |
| ✗ 有償 | そもそも無償ではない(従量課金・買い切り等) |

「ライセンス純粋」= ◎ または、軽微な条件で済む ○(クレジット表記・別プロセス配布で足りるコピーレフト)。
△・✗ は**純粋性最優先の方針に抵触**するため、採用には個別のユーザ承認が必要。

## 3. 条件付き・要注意項目の詳細

### A. Claude API は無償ではない(従量課金)

`@anthropic-ai/sdk` は MIT だが、**会話のたびにトークン課金が発生する有料クラウドサービス**。
これはライセンス問題ではなく、ビジョン上の意図的な外部依存(「外部送信は Claude API のみ」)。
「全コンポーネントが無償」という前提はここでのみ崩れる。設計の前提として明示しておく。

### B. AivisSpeech Engine は LGPL-3.0(コピーレフト)

GitHub で確認(VOICEVOX ENGINE 由来の LGPL-3.0 を単独継承)。
ene-desktop は**別プロセス(`run.exe` サイドカー)として起動し、自前コードにリンクしていない**ため、
LGPL の軽い側の義務で足りる。

- 守ること:(1) エンジンの **LICENSE 文を配布物に同梱**、(2) エンジンを**差し替え可能な形**で配る
  (改変して隠さない)。
- これらを満たせば、**アプリ本体のソースを公開する義務はない**(クローズドソース配布可)。

### C. つくよみちゃんコーパス由来の声 — 商用OK・条件付き

- 商用利用は有料/無料/広告問わず可・**売上上限なし**。
- **クレジット必須**:声質そのものを第三者が利用できる形にする場合。`voice.json` に記載済み(維持すること)。
- **禁止用途**:人を批判・攻撃する言動/特定の政治・宗教・思想への賛否を呼びかける用途。
- **コーパス本体の再配布は禁止**(配布するのは派生モデル `.aivmx` でありコーパスではないので可)。
- Vol.1 は CC BY-SA 4.0 ⇒ 派生声モデルにも **share-alike が及び得る**点に留意。

## 4. 表示キャラクターの実装方式 — ライセンス観点の選択肢

「VRM・立ち絵差分・立ち絵+微動 以外に選択肢はないのか」への回答。
**他にもある。** ライセンス純粋のまま「Live2D 風の滑らかな変形」を出せる FOSS リグ系が存在する。
ただし各々に実用上の注意点がある。

### 4.1 ライセンス純粋(売上連動なし)

| 方式 | 種別 | ライセンス | Electron親和性 | 注意点 |
|---|---|---|---|---|
| 立ち絵差分 | 2Dコマ切替 | レンダラ次第(PIXI=MIT) | ◎ | 補間なし(パッと切替)。**現行実装** |
| 立ち絵+微動 | 2D補間 | PIXI.js=MIT | ◎ | まばたき/呼吸/クロスフェード。自前実装 |
| **Rive** | 2Dステートマシン | ランタイム=MIT / エディタ無料 | ◎(WASM成熟) | エディタは非OSSだが無料・書き出し永続。便利機能で月$9誘導あるが必須でない |
| **DragonBones** | 2Dスケルタル | ランタイム=MIT | ○(DragonBonesJS+Pixi) | **開発停止**=長期保守リスク(vision「何十年後も使えるか」と相性悪) |
| **Inochi2D / nijilive** | 2Dメッシュ変形 | BSD-2 | △(Dネイティブ中心) | **思想的に最も理想**(FOSS版Live2D・PSDほぼ互換)。**β・本番非推奨/JSランタイム未成熟**で現状Electron組み込みが困難 |
| **VRM / three-vrm** | 3D | MIT/Apache | ◎ | 重い・原案イラストは流用不可(VRoidで作り直し) |
| (Lottie) | 2Dベイク済み | lottie-web=MIT | ◎ | 感情リグには非力。簡易モーション向け |

### 4.2 ライセンス非純粋(=純粋性最優先の方針では除外)

| 方式 | ライセンス | 判定 |
|---|---|---|
| **Live2D Cubism** | 年商1000万円で有償 | △ 売上連動 |
| **Spine** | 年商50万ドルで Enterprise 必須＋エディタ$70〜330 | △ 売上連動＋✗ 有償 |

### 4.3 純粋性最優先での推奨

1. **立ち絵+微動(PIXI)** — 最も単純・完全MIT・現行資産を活かせる。短期の最適解。
2. **Rive** — 滑らかな2D・感情ステートマシン・MITランタイム・成熟したWeb組み込み。
   「Live2D風の表現が欲しいが純粋無償」を最も現実的に満たす。
3. **VRM / three-vrm** — 3Dが欲しい場合。完全無償だが重さと原案作り直しがトレードオフ。

- **Inochi2D** は理想だが現時点では Electron 組み込みが困難。**将来の有力候補として継続ウォッチ**。
- **DragonBones** はMITだが保守停止のため、長期前提の本プロダクトでは非推奨。
- **Live2D / Spine** は純粋性方針に抵触するため、採用するならユーザの明示承認が必要。

## 5. 監査の結論

- 現行の依存・モデル・開発ツールは、**Claude API(従量課金)を除き、売上に関係なく無償**。
  寛容ライセンス(MIT/Apache/BSD)が中核で、ライセンス純粋性は保たれている。
- 条件付きは3点のみ:**AivisSpeech(LGPL=別プロセスで軽い義務)**・**つくよみ由来の声
  (クレジット＋禁止用途)**・**Claude API(有料)**。いずれも運用で対応可能。
- 表示方式に **Live2D / Spine** を入れると、**初めて売上連動ライセンスが混入**する。
  純粋性最優先の方針では、**立ち絵+微動・Rive・VRM** のいずれかを採るのが整合的。

## 6. 実用候補の Pro / Con(ライセンス純粋前提)

| 方式 | ライセンス | 滑らかさ | 実装難度 | 原案再現 | 軽さ | 保守安心度 |
|---|---|---|---|---|---|---|
| ① 立ち絵差分(現行) | MIT | 低(パッと切替) | 最低 | ◎ | ◎ | ◎ |
| ② 立ち絵+微動(PIXI) | MIT | 中 | 低 | ◎ | ◎ | ◎ |
| ③ Rive | ランタイムMIT/エディタ無料 | 高 | 中 | ○ | ○ | ○ |
| ④ VRM / three-vrm | MIT/Apache | 高 | 高 | △ | △ | ◎ |

**① 立ち絵差分(現行実装)**
- Pro:すでに動作。完全MITで最軽量・最小CPU。必要スキルはイラストのみ。原案の画風をそのまま使える。感情ラベル契約(`emotion`)が既に通っている。
- Con:表情・ポーズが補間なしで「パッ」と切替=生命感が弱い。表現の幅が静止画の枚数で頭打ち。

**② 立ち絵+微動(PIXI.js で自前実装)**
- Pro:①にまばたき・呼吸の上下・表情クロスフェード・視線追従を足すだけで生命感が大きく上がる。完全MITで外部ランタイム不要、軽さ維持。①の資産をそのまま拡張でき、リスク最小。
- Con:動きは「微動」止まり。口や体の本格変形は出せない。凝るほど自前のアニメ制御コードが増える。

**③ Rive**
- Pro:純粋ライセンスで「Live2D風の滑らかさ」に最も近づける現実解。感情ごとのステートマシンを視覚的に組め `emotion` ラベルと好相性。WASM/Webランタイムが成熟しElectron組み込みが容易。ランタイムMIT・書き出し永続・エディタ無料。
- Con:エディタ自体は非OSS(無料だが)。「ライブ公開」向けに月$9へ誘導される場面あり(必須でない)。原案はRive上でベクター/メッシュに作り直す手間。学習コストは②より高い。

**④ VRM / three-vrm**
- Pro:完全無償(MIT/Apache)の3D。表情はブレンドシェイプ、ポーズは骨で自由=遷移を手描き不要。カメラ角度自由・mocap/VRMA流用可。長期保守も安定。
- Con:常時最前面のWebGL常駐で最も重く「軽量100MB/CPU3%」原則と最も衝突。VRoidは原案を取り込めず作り直し=画風が変わる。実装も最重。

**総括**:確実性とリスク最小なら **②**、滑らかさを取りに行くなら **③ Rive**、3Dにこだわるなら **④**。
①→② は地続きなので、**②で土台を作りつつ③Riveを試す二段構え**が最もリスクが低い。

## 7. VRM / three-vrm を採用した場合のリソース見積もり

**前提**:背景は静止(PNG または透過)、表示は**バストアップのみ**(VTuber配信風)、Windows・単体アバター。
数値は概算レンジ(機種・テクスチャ解像度・実装最適化で変動)。

| 項目 | 見積もり | 備考 |
|---|---|---|
| exe同梱増分(three + three-vrm) | **+約1〜1.5MB**(gzip ~0.25MB) | three.js gzip ~155KB＋three-vrm。**100MB枠に十分余裕** |
| VRMモデル本体(**data/ 配置・非同梱**) | 最適化なし **~15〜18MB** / 2K最適化 **~5MB** / 下半身メッシュ削除でさらに圧縮 | whisper/ruri と同じく別アセット扱い。コア<100MBは不汚染 |
| 追加RAM(描画) | **+約150〜350MB** | Electron renderer に WebGL コンテキスト＋モデル |
| テクスチャVRAM | 2K: **~10〜30MB** / 4K: ~200MB | **2K以下推奨**(4Kはバストアップに過剰) |
| 常駐CPU(無対策・60fps) | 1コアの **~5〜15%** | **3%目標を超過する懸念=最大の論点** |
| 常駐CPU(対策後・30fps/アイドル間引き) | **~2〜5%** | fps制限＋非発話時の間引き＋非表示時停止で目標近傍まで圧縮可 |
| GPU | 単体アバターは軽い | 常時最前面の透過合成(DWM)コストは別途 |

### 7.1 結論(軽量原則との関係)

- **配布サイズは問題にならない**。JS増分は約1MBで、VRMモデルは `data/` に置けばコア100MBを汚さない(既存のモデル配布パターンと同じ・§2.3 と整合)。
- **真の制約は「常駐CPU 3%(柱4)」**。WebGLの常時描画は、PNG差し替え(ほぼ0%)より本質的に重い。**fps制限・アイドル間引き・非表示時の停止**を実装すれば目標近傍に収められるが、現行方式より確実に重くなる。RAM +150〜350MB も「軽量常駐」前提では無視できない。
- **バストアップ＋静止背景は有利**:下半身の歩行/IK不要、必要なのは「まばたき・呼吸・頭の向き(look-at)・表情ブレンドシェイプ・口形(リップシンク)」のみ。アニメ作り込みもCPUも軽くなる。ウィンドウ実描画解像度を小さく保てる点も有利。

### 7.2 軽さを保つための実装指針(採用する場合)

- テクスチャ **2K以下**・マテリアル数削減・**見えない下半身メッシュを Blender で削除** → モデル ~3〜6MB。
- 描画を **30fps上限**、非発話・無操作時は **~10fps か render-on-demand** に落とす。**ウィンドウ非表示/被覆時は描画停止**。
- 既存の **`emotion` ラベル契約をそのまま流用** → VRM標準の表情プリセット(happy/angry/sad/relaxed/surprised)へマップ。リップシンクは口形(aa/ih/ou/ee/oh)ブレンドシェイプへ。
- これらは vision 柱4・成功基準7(CPU3%/100MB)の**計測を採用判断の関門**にすること。

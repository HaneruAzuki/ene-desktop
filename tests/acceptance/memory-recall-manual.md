# 手動テスト手順:記憶の想起・更新(task_15)

> 想起エンジンが「意図通りに」動くかを、実アプリで人間が確認するための手順。
> 自動単体テスト(全パターン)は `tests/unit/` の retriever / index-inverted / index-vector /
> update / extractor-corrections / recall-update-integration / episodic-v2 を参照
> (API・実モデル不要で網羅済み)。本書は **実体験での最終確認** を担う。

---

## 0. 想定している「記憶 → 想起」プロセス(これが正しく動けば成功)

1. 会話 → 中立な要約(episodic)として `data/memory/ene/episodic/{年}/{分類}/` に保存。
   - 人物・固有名は `entities`(正規名)に、軽い語彙は `tags` に入る。
2. 次の発言時、**その発言を引き金に全件横断で想起**(Router 非依存):
   - **語彙/人物(entity)一致**:いつでも有効(モデル不要)。
   - **意味(ベクトル)一致**:`ruri` モデル配置時のみ有効(字面が違っても意味が近ければ引く)。
   - 両者を **RRF でローカル合流**。`supersededBy` が付いた古い記憶は除外。
3. 古い情報は **非破壊更新(supersede/refine/reattribute)**。物理削除はしない。

---

## 1. 準備

1. **依存導入**(初回のみ):`npm install`
2. **APIキー**:通常どおりアプリ起動時のダイアログで設定(会話に必要)。
3. **仮の記憶を投入**:`npm run seed:recall`
   - `data/memory/ene/episodic/` に5件の仮記憶が作成される(田中×2 / 勉強 / ラーメン / 鈴木)。
4. **(意味想起を試す場合)モデル配置**:`npm run download:model`
   - `data/models/ruri-v3-310m/`(約316MB)に int8 ONNX を配置。
   - **未配置でも可**。その場合は「語彙/人物想起」だけが効く(意味の橋渡しは無効)。
5. 起動:`npm run dev`

> データの中身はいつでも素のJSONで確認できる:
> `data/memory/ene/episodic/**/*.json`(各記憶)、`data/memory/ene/index/inverted.json`(語彙索引)、
> `data/memory/ene/index/vectors.json`(ベクトル索引・モデル配置時に自動生成)。

## 2. シナリオ(入力 → 期待する想起)

各シナリオは「ENEの応答が、その記憶を踏まえている」ことを人間が判定する。
(応答の口調が自然か=AIっぽくないか は `manual-check.md`/成功基準8 と同じく人間判定。)

### S1. 人物で束ねる横断想起(モデル不要)
- 入力例:「最近、田中さんとどうなんだっけ?」
- 期待:**話題の違う2件**(仕事の衝突 / カラオケで仲直り)の両方を踏まえて応答する。
  片方しか出ない/全く触れない場合は人物束ね想起が弱い。

### S2. 単純な好みの想起(モデル不要)
- 入力例:「お腹すいた…何食べようかな」
- 期待:ラーメン(味噌)が好きという過去を踏まえて応答する。

### S3. 意味の橋渡し(**モデル配置時のみ**)
- 入力例:「今回のテスト、赤点だったかも…」
- 期待:字面に「赤点」が無い「実力テスト前に遊ぶと言って勉強を反対された」記憶を引いて、
  「だから言ったでしょ!」系の反応につながる。
- モデル未配置時:語彙が一致しない(「赤点」≠「勉強/テスト」)ため引けないことがある。
  これは仕様(意味想起は要モデル)。`vectors.json` が生成されているかも確認。

### S4. 記憶の非破壊更新:supersede(要・抽出トリガ)
- 入力例(数ターン会話したうえで):「やっぱり鈴木のことは友達としてしか見れないや」
- 抽出の起こし方(どちらか):
  - そのまま会話を20往復超続ける(短期記憶 overflow で抽出)、または
  - **アプリを一度終了する(終了時に抽出が走る)→ 再起動**。
- 期待(再確認):
  - `data/memory/ene/episodic/2025/relationship/2025-11-02T22-15-00.json` に
    **`supersededBy` が付く**(旧「鈴木が好き」は残るが current から外れる)。
  - 新しい「友達として見ている」要約ファイルが追加される。
  - 以後「鈴木」の話題で、ENEが**古い恋愛前提でなく新しい認識**で応答する。
  - ※ 確信が低いと ENE は自動上書きせず、会話で確認しようとする(自動上書き禁止)。

### S5. 人物の取り違え:reattribute(要・抽出トリガ)
- 入力例:「さっき話した田中、正確には田中一郎さんのことね」
- 抽出後の期待:該当する**その1件だけ** `entities` が更新される(他の「田中」記録は不変)。
  `inverted.json` で新名義から引けるようになる。

## 3. 確認チェックリスト(結果を記録)

- [ ] S1 田中の横断想起(2件とも踏まえる)
- [ ] S2 ラーメンの想起
- [ ] S3 意味の橋渡し(モデル配置時)/ 未配置時は語彙のみで妥当
- [ ] S4 supersede が非破壊で効く(旧記憶は残り `supersededBy` 付与・新認識で応答)
